1. 前言與 MVVM 架構理念

Claw-Note 是一款功能強大的智慧型課堂講義管理應用程式。本專案以 Flutter 作為前端框架，本地端使用 SQLite 作為離線快取與樹狀檔案管理，雲端則緊密整合了 Firebase Auth、Firebase Storage 與透過 Python 編寫的 Firebase Cloud Functions 作為 AI 中樞後端。

為了確保程式碼的可維護性、測試性與職責分離，專案採用了 MVVM (Model-View-ViewModel) 的架構變體：

  • Model (資料模型/實體)：定義純粹的資料結構與序列化邏輯（如 annotation_model.dart、models.dart、ai_page_note.dart）。
  • View (視圖/畫面)：負責 UI 佈局、接收使用者手勢，不直接執行計算或編排 API Request。視圖訂閱 ViewModel 的改變並渲染（如 screens/、widgets/）。
  • ViewModel (介面狀態管理)：保存 UI 對應的狀態（如 loading、error、數據清單），繼承自 ChangeNotifier，在狀態變更時發布廣播（如 viewmodels/）。
  • Repository & Service (倉庫與服務)：ViewModel 不直接存取資料庫或網絡。ViewModel 調用 Repository 來編排業務邏輯，Repository 進一步使用具體的 Service（如 SQLite、Azure 語音、Firebase API）來處理實體資料。

2. Dart / Flutter 語法與概念大解密

在閱讀 Claw-Note 原始碼前，必須先掌握 Dart 在非同步與狀態監聽上的核心語法，這也是初學者最常感到困惑的部分。

Future, async & await

什麼是 Future？
Future<T> 代表一個「在未來才會完成的非同步計算結果」，類似於 JS 中的 Promise。它有三種狀態：未完成 (Uncompleted)、完成並回傳值 (Completed with value) 與失敗 (Completed with error)。

async & await 的功用：
async 關鍵字用來標記一個函式為非同步函式，使其內部可以使用 await。await 會「暫停」當前函式的執行，等待 Future 完成並取得值後再往下執行，而不會阻塞 Flutter 的 UI 主執行緒。 

Future<String> fetchData() async {
  // 模擬 2 秒的網絡延遲
  await Future.delayed(Duration(seconds: 2));
  return "資料載入成功";
}

Stream & 訂閱機制

什麼是 Stream？
如果說 Future 是完成一次性任務（如獲取單次 HTTP 響應），那麼 Stream 就是「源源不絕的非同步資料流」。它就像一條生產線或水管，水管的一端負責送入資料，另一端則有監聽者（Listener）負責接收。

生動比喻： Stream 就像是「即時收音機廣播」。電台源源不絕地播出聲音訊號，只要收音機開著並轉到該頻道（訂閱 listen），就能即時聽到聲音。

核心組件：

  • StreamController：控制串流的閥門。可以調用 add() 塞入新資料，其 .stream 屬性可供外部監聽。
  • StreamSubscription：訂閱物件。當呼叫 stream.listen() 時會回傳此物件。必須在元件銷毀時調用 cancel() 以免記憶體洩漏！

廣播串流 (Broadcast Stream)：
預設 Stream 只能被單一 Listener 監聽（Single-subscription）。在課堂視圖中，我們使用了 .broadcast() 建立廣播串流，這允許多個面板（如 ChatbotPanel、SummaryPanel）同時監聽同一個音訊逐字稿片段。

ChangeNotifier & ValueNotifier

ChangeNotifier：
這是 Flutter SDK 提供的一個簡單類別。它允許類別維護多個監聽者。當 ViewModel 內部狀態改變時，呼叫 notifyListeners() 會通知所有訂閱此類別的 Widget 調用 setState() 重繪 UI。 

class CounterViewModel extends ChangeNotifier {
  int _count = 0;
  int get count => _count;

  void increment() {
    _count++;
    notifyListeners(); // 廣播給所有 UI 進行重繪
  }
}

ValueNotifier：
ValueNotifier<T> 是 ChangeNotifier 的輕量級子類別，專門用於「只持有一個變數值」的場景。只要將 notifier.value 賦予新值，它便會自動呼叫 notifyListeners()。

本專案在 slides_panel.dart 與 annotation_manager.dart 中，利用 ValueNotifier<List<Annotation>> 實現了：當 BBox API 運算結束或使用者手動添加標記時，投影片繪圖 Canvas 能在毫秒內自動重繪，而無需重置整頁 UI。

unawaited 函式

為什麼要用 unawaited？
在 Dart 的非同步設計中，如果呼叫了一個回傳 Future 的函式但沒有使用 await，編譯器（linter）會發出警告，提醒你可能漏掉了等待。

但在很多場景下，我們刻意不想等待它。例如：開始錄音時，我們希望在背景非同步執行學生的專注度記錄初始化或 Firebase 備份，不希望 UI 被卡住等待。這時呼叫 unawaited(myFutureTask()) 即可明確告訴編譯器：「我知道它是非同步的，且我故意不想等待它完成」。

WidgetsBindingObserver (生命週期)

這是 Flutter 監聽 App 生命週期的介面。當用戶收到來電、切換到別的應用程式（App 進入背景），或是重新回到 Claw-Note（進入前景），Flutter 會透過此 Observer 發出通知。

本專案的 StudentAttentionTracker 繼承了它，藉此追蹤學生是否將 App 切換到背景，並將此「分心行為」傳送給後端的 Attention Agent 進行冷靜分析。

StatefulBuilder & StateSetter

在 Flutter 中，彈出視窗（如 showDialog）本質上是推入一個獨立的路由頁面。此時如果在原畫面調用 setState，彈窗內部的內容（例如備份進度條與文字）是不會更新的。

為了解決此問題，我們使用 StatefulBuilder 包裹彈窗內容。它提供了一個專屬的 StateSetter（通常命名為 setDialogState），當 ViewModel 更新備份進度時，調用 setDialogState(() {}) 就能局部且精確地刷新 Dialog 的狀態。

4. 資料夾與檔案完全手冊

本節將深入專案 lib/ 目錄下的所有子資料夾，並列出每個 .dart 檔案的詳細說明。

A. 根目錄與資料結構層 (Root & Data / Database / Models)

負責應用的初始化、全域設定、本地資料庫 schema 與純粹的資料模型實體結構。

lib/firebase_options.dart 設定/環境

核心職責： 儲存由 FlutterFire CLI 自動產生的金鑰與專案識別碼，讓 Flutter 應用程式能正確初始化 Firebase 連線。

重要屬性/類別職責與邏輯
DefaultFirebaseOptions 類別 提供 currentPlatform 靜態屬性，偵測當前作業系統（Android, iOS, macOS, Web, Windows），回傳對應的 FirebaseOptions 初始化參數。

關聯相依： 僅被 lib/main.dart 引入，用於初始化 Firebase 的核心連線。

lib/main.dart 核心入口

核心職責： 應用的啟動入口，初始化 Firebase 與本地推播通知服務，配置 GoRouter 宣告式路由導覽，並建立全域主題設定。

重要函式/變數參數與返回值職責與內部邏輯
main() Future<void> 先確保 Flutter Engine 綁定初始化，隨後以 Firebase.initializeApp() 啟動 Firebase；向 FCM 註冊背景推播處理器，最後調用 runApp(const MyApp())。
_router 變數 GoRouter 設定三層路由：/ 對應 Dashboard，/course/:courseId 對應 CourseDetails，/lecture/:courseId/:fileId 對應 LectureView。使用 ShellRoute 與 LayoutWrapper 為所有畫面提供側邊導覽。

關聯相依： 依賴 notification_service.dart。調用並連結所有頂層畫面（Screens）與佈局外殼（LayoutWrapper）。

lib/data/annotation_model.dart 資料模型

核心職責： 定義 PDF 投影片上的自訂標記（矩形標記與文字標記）。負責標記的 JSON 序列化與 Canvas 繪圖渲染。

主要類別與函式參數與返回值職責與內部邏輯
Annotation (抽象類別) 成員：id, pageIndex, type, color 標記的基底類別。包含 factory Annotation.fromJson，能根據 JSON 的 type 自動分流實例化為矩形或文字標記。定義 draw() 抽象方法。
RectAnnotation 成員：x, y, width, height (使用相對座標 0.0 ~ 1.0) 繼承自 Annotation。實作 draw() 方法，在 Canvas 上繪製矩形與半透明背景的文字標籤。相對座標乘以 Canvas 實際寬高，以適應不同解析度與縮放。
TextAnnotation 成員：text, fontSize, autoWrap 繼承自 Annotation。使用 TextPainter 在 Canvas 上繪製自訂文字標記，字型大小根據 850.0 基準寬度動態調整比例，防止縮放失真。

關聯相依： 被本地資料庫的標記保存擴充協調，以及簡報繪製視圖 slide_page.dart 使用。

lib/data/transcript_data.dart 資料模型 & 靜態資料

核心職責： 定義逐字稿頁面與章節結構，並提供 Demo 演示模式下所使用的範例硬體課程逐字稿靜態資料。

主要類別/變數類型與描述用途

chapter4_1TranscriptData List<TranscriptPage> 靜態資料 內含計算機組織第四章 Datapath 與 Control Signals 的真實中文講義逐字稿，用於模擬 Demo 模式下的逐字打字流。

關聯相依： 用於 lecture_view.dart 中的 Demo 錄製模式。

C. 核心服務層 (Services)

處理所有具體的底層技術實作，包含音訊錄製、Azure 語音辨識 WebSocket、雲端 Functions 連接、專注度事件偵測以及通知推播。

lib/services/annotation_manager.dart 服務

核心職責： 管理每頁 PDF 的標記生命週期。使用記憶體緩存與防抖動技術 (Debounce) 批次寫入資料庫，並防止在關閉時保存不完整的標記。

主要方法/邏輯參數與返回值內部工作流程與防抖機制
getPageNotifier() int pageIndex
回傳 ValueNotifier<List<Annotation>>		 如果快取中已有該頁的 Notifier 則直接返回，以達到瞬間渲染；若無，則異步自本地 SQLite 載入標記資料並回填。
_markPageAsDirty() int pageIndex 當新增/刪除標記時，將該頁標記為「髒頁 (dirty)」，取消先前的計時器，並重新設定一個 600 毫秒的計時器。600 毫秒內若無新操作，才批次寫入 SQLite，大幅降低資料庫負擔。
dispose() 無 釋放所有 Notifier。若偵測到標記產生狀態為 generating，說明是中途關閉簡報，會自動清空並丟棄已生成的暫存標記。
lib/services/auth_service.dart 服務

核心職責： 透過本地 ID Token 向 Firebase Functions azureSpeechToken 請求一個短期的 Azure Speech 驗證 Token，避免金鑰直接曝露在客戶端程式碼中。

lib/services/azure_speech_service.dart 硬體 & 網路

核心職責： 連接 Azure Speech 即時語音轉文字 WebSocket API。開啟本機 16kHz PCM 麥克風錄音，並將音訊二進位流以專用 framing 格式推送至雲端，解析 hypothesis 與 finalized 轉譯字元。

Azure 封裝協議關鍵： Raw PCM 音訊流在發送給 Azure 之前，第一個二進位封包必須包含 RIFF WAV 標頭（_buildWavHeader()），否則 Azure 會拒絕解析！此外，每個語音封包都必須加上自訂的文字標頭，格式為： 
Path: audio\r\nX-RequestId: {ID}\r\nX-Timestamp: {TIME}\r\nContent-Type: audio/x-wav\r\n\r\n{二進位音訊數據}
主要成員與方法參數與返回值職責與內部邏輯
startListening() String token 請求麥克風權限。產生唯一的 RequestId，建立 Azure WebSocket。連線成功後，先發送 speech.config JSON 配置（設定目標語言如 zh-TW），再啟動 record 串流，並訂閱音訊流將數據打包送出。
_handleAzureResponse() dynamic event 解析 Azure 傳回的字串。若包含 speech.hypothesis 代表暫時性結果，更新 UI 顯示；若包含 speech.phrase 代表該句結束，將其轉為 TranscriptChunk 寫入本地，並調用 saveTranscript() 存檔。
transcriptStream Stream<String> (Broadcast) 供 UI 訂閱，即時將已識別文字呈現在 TranscriptPanel 上。
lib/services/bounding_box_service.dart 服務

核心職責： 與外部 Python Cloud Run 圖像偵測 API 通訊。上傳投影片圖片，偵測圖片中的元件邊界座標（X1, Y1, X2, Y2）與色彩資訊。

主要方法參數與用途端點說明
fetchAnnotatedPipeline() List<int> imageBytes
回傳所有偵測到的標籤與矩形坐標		 POST 請求至 /detect/pipeline。用於初始化的簡報投影片自動標記定位。

核心職責： 全局管理課堂簡報摘要（Notes）的生成流程與實時更新。當簡報重新生成時，負責將舊有的「實時課堂筆記」與新生成的「投影片摘要」進行防丟失去重合併。

主要方法參數與用途職責與去重合併演算法
generate() storageId, pdfPath, courseId, lectureId 向 Firebase Storage 上傳簡報 PDF，隨後調用 generateNotesFromPdf 雲端函式。取得筆記 JSON 後與本地舊筆記合併，寫入本地 notes.json 與 notes/page_NNN.md。
appendRealtimeUpdate() storageId, pageNumber, newPoints, questions 當實時代理分析出老師有新的板書重點時，在該頁 Markdown 末尾追加 ### Professor Additions 與 ### Professor Questions 區段。會先解析舊有文字，進行大小寫無關的重複行去重後再行追加，防止重複寫入。
_mergeRealtimeUpdates() generatedNotes, currentNotes 防遺失演算法： 重新生成筆記時，先掃描現有筆記中是否含有 Live Lecture Updates 等實時追加區段。若有，則提取該區段，並在雲端生成新摘要完成後，自動將實時段落附加到新摘要末尾，確保學生在課堂上記錄的板書不會被覆寫消失。
lib/services/notification_service.dart 服務

核心職責： 初始化並請求本機通知權限。向 Firebase Messaging 獲取 FCM Token，並訂閱雲端推播。當接收到類型為 attention_distraction 的推播時，代表 Attention Agent 判斷學生已分心，即時彈出本機通知提醒學生老師目前的頁碼。

lib/services/realtime_agent_coordinator.dart 核心協調器

核心職責： 實時課堂的核心協調中樞。監聽每 10 秒產生的逐字稿片段，使用佇列 (Queue) 依序調用後端 realtimeAgent Functions 判定老師頁碼，並動態分派任務去更新 AI 摘要或在投影片畫板上框選重點。

資訊隔離安全機制： 協調器向後端傳送的狀態中，絕對不會包含學生當前看見的頁面。後端 Realtime Lecture Agent 僅基於逐字稿與既有頁面內容判斷老師所在頁面。判斷完成後，後端才結合學生狀態進行 Attention 分心檢測。這防止了 Realtime Agent「作弊偷看」學生的頁面。
重要成員與方法職責與工作隊列 (Queue) 排水邏輯
_subscribe() 監聽 segmentStream。將非空片段包裝為 _QueuedTranscriptChunk（包含當前片段與最近 9 個歷史片段組成的滾動窗口），存入 _pendingChunks 佇列，隨後調用 _drainQueue()。
_drainQueue() 防併發排水鎖。若目前有任務正在運行，則直接返回。否則啟動 while 迴圈，逐一將佇列首部的片段移出並傳送至後端 Functions，保證異步網路請求順序執行且不漏失。
_processTranscriptChunk() 發送 POST 至 realtimeAgent。取得回傳後，如果 update_note_at 為 summary，調用 notesViewModel 寫入板書追加；若為 slides，則調用 slidesViewModel 動態發送影像進行特定目標框選。呼叫 _handleAttention 評估是否需要透過 NotificationService 彈出通知。
lib/services/student_attention_tracker.dart 專注度追蹤

核心職責： 追蹤學生當前觀看的頁面、停留時間與應用程式的前背景狀態。實作 WidgetsBindingObserver，將學生的閱讀事件寫入 SQLite student_page_events 表，並產生快照供實時代理使用。

主要方法職責與進入/離開時間計算
start() 與 stop() 開始/結束追蹤。向 currentPageNotifier 註冊監聽器，並向 Flutter 引擎註冊生命週期觀察者。
_handlePageChanged() 當學生滾動投影片使頁碼改變時，計算當前時間與 _currentPageEnteredAt 的差值，作為停留秒數。將此事件寫入 SQLite，並更新當前頁碼與新一頁的進入時間。
snapshot() 格式化輸出目前的追蹤快照。包含當前頁碼、在當前頁停留的秒數、最近 20 次頁面歷史、App 是否在前景、以及退到背景的時間點。
lib/services/transcript_export_service.dart 資料導出

核心職責： 每 10 秒導出一次逐字稿片段。比較本次與上次的文本，計算出「新增的差值 (DeltaText)」，寫成本地 seg_NNN.json 段落檔，更新本地 SQLite 錄音記錄，並以廣播串流發送片段。

核心演算法邏輯與 empty 佔位
exportSegment() 計算增量：
1. 若當前全文以上次全文開頭且長度增加，差值即為 substring(lastText.length)。
2. 若完全沒有變化，差值為空字串，且 JSON 欄位 is_empty 設為 true。
將 payload 寫入本地 transcripts 資料夾下該錄音 session 子目錄內。呼叫 onSegmentExported 廣播回呼，並將全文更新到 SQLite items 表該 recording 節點中。
lib/services/user_identity_service.dart 服務

核心職責： 確保使用者登入。使用 Firebase Auth 的 signInAnonymously() 進行匿名登入，並返回使用者物件。這允許使用者即開即用，且能保持各個使用者在 Firestore 中的 Long-term Memory 數據隔離。

5. 核心端到端資料流說明

為方便您串接與理解，以下詳細解密專案中三大最關鍵的 AI 資料流運作順序：

1. 簡報上傳與 AI 筆記生成資料流

  1. 使用者在 SlidesPanel 上傳簡報 PDF。
  2. SlidesPanel 將 PDF 複製到本地 App 儲存路徑，並回寫 SQLite filePath。
  3. 呼叫 LectureNotesViewModel.generateFromPdf()，其底層調用全域單例 NoteGenerationManager。
  4. NoteGenerationManager 先將 PDF 上傳到 Firebase Storage (路徑：ai_note_jobs/{safeStorageId}/source/...)。
  5. 上傳完成後，調用 FirebaseFunctionClient.postJson() 呼叫雲端 generateNotesFromPdf 函數，附帶當前使用者的 ID Token 進行身分驗證。
  6. 雲端處理中 (最長 10 分鐘)：Functions 偵測用戶在 Firestore 的學習偏好與 Memory，讀取 PDF，調用 OpenAI 產生逐頁 Markdown 摘要。
  7. Functions 生成完畢後，將結果寫回 Firebase Storage，並在 HTTP 回應中返回 JSON 資料。
  8. NoteGenerationManager 接收到回應後，讀取舊筆記中的板書更新進行防遺失合併 (Cosine Deduplication Merge)。
  9. 將最終合併的筆記寫入本地 notes.json 與 notes/page_NNN.md。
  10. LectureNotesViewModel 透過 Watch Stream 接收到 Completed 狀態，通知 SummaryPanel 渲染 Markdown。

2. 課堂錄音、實時老師定位與板書追加資料流

  1. 使用者在 TranscriptPanel 點擊「開始錄音」（或開啟 Demo Mode）。
  2. 應用程式透過 AzureSpeechService 啟動麥克風，直接建立與 Azure Speech 的 WebSocket 連線。
  3. 即時語音辨識結果透過 transcriptStream 廣播至 UI，即時呈現在 TranscriptPanel。
  4. 同時，LectureView 啟動一個 10 秒的計時器，定期調用 TranscriptExportService.exportSegment()。
  5. TranscriptExportService 計算這 10 秒內新增的差值文字 (DeltaText)，寫入本地 seg_NNN.json。
  6. 導出的 JSON 物件被送入 segmentStream。
  7. RealtimeAgentCoordinator 監聽該 Stream，將非空片段存入排水佇列 (Queue)。
  8. 佇列依序調用雲端 realtimeAgent 函式，傳入近 10 份逐字稿片段與既有簡報摘要。
  9. AI 代理分析：後端的 Realtime Lecture Agent 判定老師此時正在講第 X 頁，並判定是否需要更新摘要。
  10. 如果需要更新，回傳結構化的板書重點 (new_points) 與提問 (questions)。
  11. RealtimeAgentCoordinator 取得結果後，調用 LectureNotesViewModel.appendRealtimeUpdate() 寫入本地，SummaryPanel 即時渲染出「### Professor Additions」與「### Professor Questions」。

3. 專注度偵測與 FCM 通知推播資料流

  1. 在錄音啟動時，StudentAttentionTracker 開始追蹤。
  2. 當學生滾動簡報頁碼、或將 App 切換到背景時，Tracker 計算停留秒數並寫入 SQLite student_page_events。
  3. 在上述「實時老師定位流程」中，RealtimeAgentCoordinator 在發送請求時，會調用 Tracker.snapshot() 將學生的狀態一併上傳，同時附帶目前裝置的 notificationToken。
  4. 後端 realtimeAgent Functions 在分析出老師所在頁面後，將老師頁面與學生狀態快照輸入給 Attention Agent。
  5. 分心判定：若偵測到學生停留頁碼與老師講授頁碼嚴重偏離，且學生已停留該頁超過 30 秒，則判定為 distracted（分心）。
  6. 後端調用 Firebase Admin SDK，將分心事件寫入 users/{uid}/attention_events/。
  7. 同時，利用 Admin SDK 向裝置的 notificationToken 發送一個 Push Notification。
  8. 學生的手機接收到 FCM 推播，NotificationService 攔截推播並彈出 Local Notification：「你似乎暫時離開了課程，老師目前正在講授第 X 頁。」提醒學生專注。
