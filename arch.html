AI Lecture Note 架構導覽

依 2026-06-14 的 lib/、functions_python/ 與 Firebase 設定整理。預估閱讀時間約 10 分鐘。

閱讀指南

Claw-Note 是 Flutter 講義管理 App。使用者可以建立課程與筆記、匯入 PDF、標記投影片、錄製或模擬即時逐字稿、產生逐頁 AI 筆記，並以講義內容和逐字稿向 chatbot 提問。 整體採用接近 MVVM 的分層：Widget 負責顯示，ViewModel 管理畫面狀態，Repository 組合使用案例，Service 處理 SQLite、檔案、Firebase、OpenAI 與 Azure Speech。

3. 本機資料

SQLite：lecture_system_v2.db，schema version 3

Table重要欄位用途
items id、parentId、type、name、content、filePath、cloudPath、createdAt 一張 self-reference 表表示整棵檔案樹。刪除父節點時，子節點透過 foreign key cascade delete。
conversations id、courseId、createdAt 聊天 session。程式中的參數有時命名為 notebookId，實際 FK 指向 items.id。
messages conversationId、role、content、sequenceNumber、createdAt 依順序儲存 user／assistant 訊息，隨 conversation cascade delete。

第一次建立資料庫時會建立根節點 Database 與 Temp。建立一個 course 時，會自動在課程下建立 Recordings 與 AI notes 系統資料夾。常見 items.type 包含 system_folder、folder、course、notebook、recording、ai_note。

標記資料

PDF 標記沒有獨立資料表。PageAnnotationManager 將每頁的 rectangle/text annotations 序列化成 JSON，存回 SQLite 的節點內容； ValueNotifier 提供畫面即時更新，dirty pages 再批次寫回資料庫。

App documents

路徑內容
lecture_slides/從 file picker 匯入後複製到 App 管理範圍內的 PDF。
transcripts/{storageId}.jsonAzure 辨識的完整 TranscriptChunk 陣列，用於重開 lecture 時載入。
transcripts/lecture_YYYYMMDD_HHMMSS/seg_NNN.json錄製 session 每 10 秒輸出的增量逐字稿片段。
ai_notes/{storageId}/notes.json完整逐頁 AI notes JSON。
ai_notes/{storageId}/notes/page_NNN.md每頁各自一份 Markdown，方便顯示或單獨使用。
upload_metadata.jsonFirebase 備份時的 local file path → MD5 對照。
SQLite 保存索引、階層、文字內容與 cloud path；較大的 PDF、逐字稿分段和 Markdown 檔案保存在 App documents。

4. 即時逐字稿、Demo 與 segment stream

正式語音辨識

  1. LectureView 呼叫 AzureAuthService.getTemporaryToken()。
  2. AzureAuthService 透過 FirebaseFunctionClient 呼叫 azureSpeechToken。
  3. App 取得短期 token 後，AzureSpeechService 開啟 Azure Speech WebSocket。
  4. record 套件擷取 16 kHz、16-bit、mono PCM；第一個 frame 加入 WAV header，再以 Azure binary framing 傳送。
  5. Azure 的 hypothesis 更新 partial text；phrase 成功時建立 finalized TranscriptChunk 並保存完整逐字稿。

支援的 UI locale 為中文與英文；Service 會將 zh_TW／en_US 轉成 Azure 使用的 zh-TW／en-US。

Demo Mode

Transcript 面板可切換 Demo Mode。啟動後不要求麥克風，也不連 Azure，而是讀取 lib/data/transcript_data.dart 的範例講稿， 約每 100 ms 增加一個中文字元或英文 token，模擬即時辨識。Demo 仍呼叫相同的 TranscriptExportService.tick()，所以分段檔案與 stream 行為和正式錄音一致。

每 10 秒片段

開始錄音或 Demo 時，LectureView 建立 TranscriptExportService 與 10 秒 periodic timer。 每次 exportSegment() 只取「上次輸出後新增的文字」，寫成以下 JSON；即使這 10 秒沒有新文字，也會寫一個 is_empty: true 的片段。 

{
  "timestamp": "2026-06-14T06:20:00.000Z",
  "duration_seconds": 10,
  "segment_index": 3,
  "text": "本段新增的逐字稿",
  "is_empty": false
}

寫檔後，Service 透過 onSegmentExported callback 將片段加入 broadcast segmentStream。 LectureView 已把這個 stream 傳給 Slides、Summary、Chatbot，讓未來 agent 可以在課程進行中逐段反應。

目前邊界：三個面板已宣告並接收 segmentStream，但面板內尚未呼叫 listen() 或使用 StreamBuilder，所以現在不會自動摘要、更新 chatbot context 或切換投影片。真正已運作的是分段寫檔與 stream 廣播。

5. Firebase 架構

Firebase 在 main.dart 由 DefaultFirebaseOptions.currentPlatform 初始化。Flutter client 目前直接使用 firebase_core 與 firebase_storage；Firestore 只由 Cloud Functions 的 Admin SDK 操作，Flutter 端沒有 cloud_firestore dependency。

目前所有 Flutter 平台與 Firebase CLI 的預設 project 都是 ai-notes-555a6，Storage bucket 為 ai-notes-555a6.firebasestorage.app。

Firebase Storage

路徑格式寫入者內容與 metadata
ai_note_jobs/{safeStorageId}/source/{timestamp}_{fileName} Flutter NoteGenerationManager 等待 AI 處理的 PDF。Content-Type 為 application/pdf，custom metadata 包含原始 storageId 與 sourceFileName。
ai_note_jobs/{safeStorageId}/notes/notes.json generateNotesFromPdf Function OpenAI 產生的 { pages: [...] }。Content-Type 為 application/json，metadata 包含 storageId。
users/{userId}/referenced_files/{nodeId}_{fileName} Flutter FirebaseUploadService Dashboard 備份的本機實體檔案。資料夾節點不會上傳；目前 DashboardViewModel 的預設 userId 是 jenny。

Dashboard 備份會計算 MD5，和 upload_metadata.json 比對；hash 相同且 SQLite 已有相同 cloudPath 就跳過。 上傳成功後，Storage object path 會回寫 items.cloudPath。目前程式只上傳有 filePath 的節點，沒有實作還原流程，也沒有實際上傳 SQLite database file。

Cloud Firestore

generateNotesFromPdf 會更新 ai_note_jobs/{safeStorageId}，主要提供後端工作紀錄：

欄位寫入時機說明
status開始／成功／失敗running、completed、failed。
pdfStoragePath開始輸入 PDF 的 Storage path。
notesStoragePath成功輸出 notes JSON 的 Storage path。
pageCount成功產生的頁數。
error失敗Function 捕捉到的錯誤訊息。
updatedAt每次更新Firestore server timestamp。

Flutter 不讀取或監聽這份 Firestore document；完成訊號來自同一個長 HTTP response。Firestore 在目前流程中主要用於後端狀態與錯誤紀錄。

Cloud Functions

Function資源設定責任
chat 120 秒、512 MB 接收 notes、transcript、history、question，建立學習助理 prompt，呼叫 OpenAI Chat Completions，回傳 { answer }。
generateNotesFromPdf540 秒、1 GB同步 HTTP Function。下載 Storage PDF、呼叫 OpenAI、保存 notes JSON 與 Firestore 狀態，完成後在同一個 response 回傳 pages。
azureSpeechToken 60 秒、256 MB 使用後端的 AZURE_SPEECH_KEY 向 Azure issueToken endpoint 換取短期 token。Azure key 不需要進入 Flutter bundle。

三個 Functions 都已以 Python 3.13 部署到 ai-notes-555a6。azureSpeechToken 已部署，但 project 尚未設定 AZURE_SPEECH_KEY，因此呼叫時會回傳設定錯誤。

Functions 使用 Python 3.13 與 Firebase Functions 2nd Gen，region 由 FUNCTION_REGION 控制，預設 us-central1。 OpenAI model 可由 OPENAI_CHAT_MODEL、OPENAI_NOTE_MODEL 或 OPENAI_MODEL 覆寫，預設為 gpt-4o-mini；Azure region 預設 eastasia。

Flutter 如何呼叫 Functions

FirebaseFunctionClient 使用一般 HTTP POST，而不是 cloud_functions plugin。預設 URL 為 https://{region}-{projectId}.cloudfunctions.net/{functionName}，亦可用 Dart define 覆寫 base URL、project、region 或個別 Function URL，方便 emulator 與部署環境切換。

安全現況：HTTP endpoints 都允許 Access-Control-Allow-Origin: *，目前沒有 Firebase Auth、App Check 或 request-level authorization。正式公開前應限制呼叫來源、加入身分驗證與配額／濫用防護。OpenAI 與 Azure secrets 只應存在 Functions 環境設定。

6. 端到端資料流

PDF → AI notes

  1. SlidesPanel 將 PDF 複製到 lecture_slides/ 並保存 filePath。
  2. LectureNotesViewModel 呼叫 App-level singleton NoteGenerationManager。
  3. Manager 上傳 PDF，並把實際 storageBucket 與 object path 傳給長時間 generateNotesFromPdf request，避免 App 與 Function 使用不同 default bucket。
  4. LectureView 關閉時只移除 ViewModel 的 stream subscription；Manager 與 HTTP client 不會 dispose，因此 App process 存活期間 request 會繼續。
  5. Function 完成後 Manager 保存本機 notes.json 與逐頁 Markdown，再廣播 completed 狀態。
  6. 重進 Lecture 時會讀取 Manager 現有狀態；新結果成功前保留舊摘要。

問題 → Chatbot 回答

  1. ChatRepository 載入或建立 conversation，先把 user message 寫入 SQLite。
  2. 組合 AI notes、目前逐字稿、recent history 與 question。
  3. chat Function 呼叫 OpenAI 並回傳答案。
  4. assistant message 寫入 SQLite，ChatViewModel 通知 UI 更新。

麥克風 → 即時逐字稿

  1. App 從 Function 取得 Azure 短期 token。
  2. App 直接將 PCM audio 送往 Azure WebSocket。
  3. partial/final transcript 經 stream 更新 LectureView。
  4. 完整 chunks 寫本機 JSON；每 10 秒增量寫 session segment。
  5. segment 同時送入 broadcast stream，預留給面板／agent 使用。

Dashboard → Firebase 備份

  1. 讀取 SQLite 全部 items。
  2. 排除 system_folder、folder、course 與沒有 filePath 的節點。
  3. 檔案不存在則跳過；MD5 未變更則跳過上傳。
  4. 檔案上傳 referenced_files，回寫 SQLite cloudPath。
  5. ViewModel 以 progress callback 更新備份 dialog。

8. 現況與設計邊界

  • Local-first 課程樹、聊天與主要索引以 SQLite 為準；Firebase 不是完整同步資料庫。
  • No push 專案沒有 firebase_messaging 或 local notification；目前「通知」是 UI SnackBar 與 Dart stream 事件。
  • Stream ready segment stream 的 producer 已完成，但 Slides／Summary／Chatbot 的 consumer 邏輯尚未實作。
  • App process PDF notes request 可跨 Lecture 頁面繼續，但仍依賴 Flutter process；App 被關閉或系統終止時無法繼續或恢復。
  • Backup only Firebase 備份沒有 restore、刪除同步、衝突解決或多使用者登入；預設 userId 仍是固定字串。
  • Security Functions endpoint 尚未驗證呼叫者；若進入正式環境，Auth／App Check／Storage Rules／Firestore Rules 應視為必要工作。
維護架構文件時，請直接修改既有模組描述。例如語音流程改變，就更新「逐字稿流程」與相關檔案索引；只有新增全新子系統時才增加新章節，避免文件變成按日期堆疊的 changelog。
